# 四阶段深度开发工作流 (Deep Dev)

## 前置依赖检测（首次运行时自动执行）

<DEPENDENCY-CHECK>
在开始工作流之前，你**必须**检测以下依赖插件是否已安装。按顺序检查：

使用 AskUserQuestion 询问用户：

- 选项 2：**跳过** — 继续不安装，部分增强功能不可用

记录检测结果并宣告：

```

```

引擎模式描述：

- 没有 Codex：`Claude Code 单引擎回退模式`
</DEPENDENCY-CHECK>

## 概述

这是一个纪律严明的四阶段工作流，核心原则：**在写代码之前，必须先审查和批准一份详细的书面计划。**

把思考和打字分开。所有创意工作发生在计划阶段，实现阶段应该是无聊的机械执行。

**双引擎编排：** 研究规划阶段通过 Codex 官方插件（`codex:rescue`）执行，编码实现由 Claude Code 执行，测试验证再交回 Codex。各取所长。

**启动时宣告：** "正在使用 deep-dev 工作流（Codex 插件 + Claude Code 双引擎模式）。"

## 适用场景

- 新功能开发（涉及多文件、多模块）
- 复杂 Bug 修复（需要深入理解系统）
- 代码重构（影响面广）
- 系统集成（需要理解现有架构）

## 不适用场景

- 单行修复、拼写错误
- 简单的配置修改
- 已有明确计划的执行任务

## 文档存放路径

所有文档存放在 `ycloud-ai-planning/inner-loop/` 下，按服务和业务分类：

```
ycloud-ai-planning/inner-loop/EPIC-<服务名>-<功能名>/
├── goal.md           # 目标描述（可选，如已有则复用）
├── research.md       # 阶段一产出：深度研究报告（Codex）
├── plan.md           # 阶段二产出：实现计划（Codex + 用户注释循环）
├── test-cases.md     # 阶段三产出：测试用例（Codex，基于 plan.md 生成）
├── status.md         # 进度状态跟踪
├── task.md           # 任务分解
├── review.md         # 阶段四产出：审核文档清单（Codex，基于 plan.md + 代码 + test-cases.md）
└── verification.md   # 阶段五产出：测试验证报告（Codex，审核确认后执行）
```

命名示例：
- `EPIC-attila-flow-trigger-refactor/`
- `EPIC-whatsapp-template-management/`
- `EPIC-datacenter-billing-optimization/`

## 流程检查清单

你必须按顺序完成以下每一步：

1. **[Codex] 深度研究** — 通过 `codex:rescue` 调用 Codex 深入阅读代码，产出 research.md
2. **[用户] 审查研究** — 用户确认研究结果的准确性，纠正误解
3. **[Codex] 撰写计划** — 通过 `codex:rescue` 调用 Codex 基于研究产出 plan.md
4. **[用户] 注释循环** — 用户在 plan.md 中添加注释，Codex 更新计划（1-6 次）
5. **[用户] 最终批准** — 计划完美后，用户确认可以实现
6. **[Codex + Claude Code] 生成测试用例 + 全量实现（并行启动）**
   - 6a. **[Codex]** 基于 plan.md 生成测试用例，产出 test-cases.md（`--background` 后台执行）
   - 6b. **[Claude Code]** 同时开始执行计划，边做边标记完成
   - 6c. **[Claude Code]** 实现过程中任何确认的改动，同步更新 plan.md 和 test-cases.md
7. **[Codex] 生成审核文档** — 实现完成后，通过 `codex:rescue` 基于最终 plan.md、实际代码变更和 test-cases.md 生成审核文档清单 review.md
8. **[用户] 审核确认** — 用户审查 review.md，确认所有审核项通过后批准进入验证阶段
9. **[Codex] 测试验证** — 通过 `codex:rescue` 基于 test-cases.md 和 review.md 进行验证、编译检查、代码审查

<HARD-GATE>
在用户明确批准计划之前，绝对不能写任何实现代码。
注释循环中 Claude 只能更新计划文档，不能实现。
</HARD-GATE>

## Codex 调用方式

所有 Codex 调用统一通过官方插件 `codex:rescue` 技能执行。

**调用模式：**

| 任务类型 | 调用方式 | 说明 |
|----------|----------|------|
| 研究（只读） | `codex:rescue` 不加 `--write` | Codex 只读代码，不修改文件 |
| 计划（只读） | `codex:rescue` 不加 `--write` | Codex 只产出计划文档 |
| 验证（只读） | `codex:rescue` 不加 `--write` | Codex 只做检查和审查 |
| 继续上一次任务 | `codex:rescue --resume` | 继续上一个 Codex 线程 |
| 指定模型 | `codex:rescue --model spark` | 使用 spark 模型（更快） |
| 后台执行 | `codex:rescue --background` | 后台运行，不阻塞 Claude Code |

**注意：** `codex:rescue` 默认会加 `--write` 标志。对于研究、计划、验证等只读任务，需要在 prompt 中明确要求 Codex **只做分析和文档输出，不修改项目代码文件**。

**回退模式（无 Codex）：** 当 Codex 不可用时，所有标记为 `[Codex]` 的步骤改用 Claude Code 的 Explore Agent（`subagent_type=Explore`，设为 `very thorough`）执行等价任务。

## 阶段一：深度研究 (Research) — Codex 执行

**执行引擎：** Codex 官方插件（`codex:rescue`）/ Claude Code Explore Agent（回退）

**目标：** 深入理解现有代码库，产出可审查的研究文档。

### 研究策略选择

根据研究范围大小，选择不同策略：

| 场景 | 策略 | 说明 |
|------|------|------|
| 单服务、聚焦模块 | **标准模式** | 一次 Codex 调用 |
| 单服务、多模块 | **聚焦子任务模式** | 拆成 3-5 个聚焦问题，依次调用（`--resume` 续接） |
| 多服务 | **混合模式** | Codex 聚焦后端 + Claude Code Explore 聚焦前端 |

### 策略一：标准模式（单服务小范围）

通过 `codex:rescue` 调用，prompt 示例：

```
codex:rescue 深入研究 <项目目录> 中的 <模块/类名> 及其相关的业务流程。

要求：
1. 只做代码分析，不修改任何文件
2. 分析系统架构、核心模块、数据流、现有约定
3. 将研究报告按以下结构输出，写入 <EPIC目录>/research.md：
   - 系统概览
   - 核心模块分析
   - 数据流
   - 现有约定和模式
   - 关键发现
   - 约束和依赖
```

### 策略二：聚焦子任务模式（推荐用于大范围研究）

**核心思想：** 将一个大研究任务拆成多个聚焦问题。

**步骤：**

1. **Claude Code 先快速扫描代码结构**（用 Glob/Grep），确定关键文件列表
2. **拆分成 3-5 个聚焦问题**
3. **第一个问题用 `codex:rescue --fresh`，后续问题用 `codex:rescue --resume` 续接同一线程**
4. **最终结果合并写入 research.md**

**示例：研究充值功能可拆分为：**

```
# 第一次调用
codex:rescue --fresh 分析 FinanceController 中充值和扣费的所有 API 端点、参数、返回值。只做分析，不修改文件。

# 续接调用
codex:rescue --resume 继续分析 BalanceService 余额增减操作的实现、事务处理、SQL。

# 再续接
codex:rescue --resume 继续分析充值记录和扣费记录的数据模型、生成逻辑。最后将所有分析结果整合写入 <EPIC目录>/research.md。
```

### 策略三：混合模式（多服务研究）

当需要研究多个服务时，**同时使用两个引擎**：
- **Codex 插件** → 研究后端 Java 服务（擅长代码深读）
- **Claude Code Explore Agent** → 研究前端 TypeScript/React（擅长快速扫描）

**Claude Code 操作流程：**
1. 启动 `codex:rescue --background` 研究后端
2. 同时启动 Explore Agent 研究前端
3. 等待两者完成，合并结果写入 research.md

### 回退机制

**当 Codex 失败或不可用时，自动切换到 Claude Code：**

1. Codex 调用失败但有部分输出 → 使用部分结果 + Claude Code 补充
2. Codex 完全失败 → Claude Code Explore Agent 接管全部研究
3. Codex 未安装或未认证 → 提示用户运行 `/codex:setup`

回退时 Claude Code 应使用 **Explore Agent**（`subagent_type=Explore`，设为 `very thorough`）执行等价的研究任务，并将结果写入相同的 research.md。

### 研究文档结构

```markdown
# [主题] 研究报告

## 系统概览
[整体架构描述]

## 核心模块分析
[每个关键模块的详细分析]

## 数据流
[数据如何在系统中流动]

## 现有约定和模式
[代码库中已建立的模式]

## 关键发现
[重要的发现、潜在问题、需要注意的点]

## 约束和依赖
[技术约束、外部依赖、不可变的接口]
```

**交付：** 将 research.md 展示给用户，等待审查和纠正。

**用户审查要点：**
- Claude 是否正确理解了系统？
- 有没有遗漏的关键部分？
- 有没有误解需要纠正？

> 这个阶段通常耗费较少 token，但能预防后续最昂贵的失败模式。

## 阶段二：计划 + 注释循环 (Planning + Annotation Cycle) — Codex 执行

### 2a：撰写计划（Codex 执行）

**执行引擎：** Codex 官方插件（`codex:rescue`）

**Claude Code 操作：** 用户审查完 research.md 后，调用：

```
codex:rescue 基于 <EPIC目录>/research.md 的研究成果，为 <功能描述> 撰写详细的实现计划。

要求：
1. 只做计划，不修改项目代码文件
2. 将计划写入 <EPIC目录>/plan.md
3. 计划需包含：目标、实现方案、代码变更（每个文件的具体修改）、将被修改的文件列表、考虑事项和权衡
4. 先不添加 TODO 清单，等注释循环完成后再添加
```

**目标：** 基于研究成果，产出详细的实现计划文档。

**要求：** 使用用户自己的 markdown 文件（plan.md），而不是 Claude Code 内置的计划模式。因为 markdown 文件是可编辑的、持久化的工件。

**计划文档结构：**

```markdown
# [功能名称] 实现计划

## 目标
[一句话描述要构建什么]

## 实现方案
[详细的实现方法解释]

## 代码变更
[每个需要修改的文件，附带代码片段]

### 文件：path/to/file.java
[具体的代码变更和新增内容]

## 将被修改的文件
- `path/to/file1.java` — 修改原因
- `path/to/file2.java` — 修改原因

## 考虑事项和权衡
[技术权衡、替代方案、风险点]

## TODO 清单
[分阶段的详细任务列表 — 最后一轮注释循环后添加]
```

**参考实现技巧：** 如果用户提供了参考代码或开源实现，优先基于参考来设计方案，而不是凭空设计。Claude 从具体参考工作得更好。

### 2b：注释循环（最核心的部分）

**这是整个工作流中用户添加最多价值的环节。**

**循环流程：**

1. Codex 写出 plan.md
2. 用户在编辑器中打开 plan.md
3. 用户直接在文档中添加内联注释（用 `<!-- 注释内容 -->` 或直接写在旁边）
4. 用户告诉 Claude："我添加了一些注释，处理所有注释并更新文档。**不要实现。**"
5. Claude 通过 `codex:rescue --resume` 让 Codex 更新计划（仅更新计划，不写代码）
6. 重复步骤 2-5，通常 1-6 次，直到计划完美

**注释类型示例：**

- 领域知识注入：`<!-- 用 drizzle:generate 做迁移，不要用原生 SQL -->`
- 纠正错误假设：`<!-- 不对，这应该是 PATCH，不是 PUT -->`
- 拒绝提议：`<!-- 删掉这整个部分，这里不需要缓存 -->`
- 重定向方案：`<!-- 这个字段应该在列表级别，不是在单个项目上 -->`
- 简短指令：`<!-- not optional -->` 或 `<!-- 用现有的 XxxService -->`

**为什么注释循环有效：**

- Markdown 文件作为**共享可变状态**，用户可以在问题的确切位置写下纠正
- 不会丢失上下文（对比聊天消息的线性流动）
- 用户注入自己的判断、产品优先级和工程权衡

### 2c：生成 TODO 清单

在最后一轮注释循环完成后，要求在计划末尾添加详细的分阶段 TODO 列表：

```markdown
## TODO 清单

### 阶段 1：[阶段名称]
- [ ] 任务 1：具体描述
- [ ] 任务 2：具体描述

### 阶段 2：[阶段名称]
- [ ] 任务 3：具体描述
- [ ] 任务 4：具体描述
```

这个清单在实现期间充当进度跟踪器。

**交付：** 展示最终计划给用户，等待明确的"可以实现"批准。

## 阶段三：测试用例生成 + 全量实现 — 并行启动

### 3a：生成测试用例（Codex 后台执行）

**执行引擎：** Codex 官方插件（`codex:rescue --background`）

**触发时机：** 用户批准计划后，在开始实现的**同时**后台启动。

**Claude Code 操作：**

```
codex:rescue --background 基于 <EPIC目录>/plan.md 的实现计划，生成详细的测试用例文档。

要求：
1. 只做测试用例设计，不修改项目代码文件
2. 将测试用例写入 <EPIC目录>/test-cases.md
3. 测试用例需覆盖：
   - 每个 TODO 任务对应的验收条件
   - 正常路径测试
   - 异常路径和边界条件测试
   - 数据完整性验证
   - 接口兼容性验证
4. 按 plan.md 中的阶段分组
```

**测试用例文档结构：**

```markdown
# [功能名称] 测试用例

## 测试范围
[基于 plan.md 的实现范围]

## 阶段 1：[阶段名称]

### TC-1.1：[测试用例名称]
- **对应任务：** TODO 任务 X
- **前置条件：** [测试前需要满足的条件]
- **测试步骤：** [具体操作步骤]
- **预期结果：** [期望的结果]
- **验证方式：** [编译检查 / 单元测试 / 集成测试 / 手动验证]
- **状态：** [ ] 未验证

### TC-1.2：[异常路径测试]
...

## 阶段 2：[阶段名称]
...

## 变更日志
[记录实现过程中对测试用例的修改]
- [日期] 新增 TC-X.X：原因
- [日期] 修改 TC-X.X：原因
```

### 3b：全量实现（Claude Code 执行）

**执行引擎：** Claude Code

**前提：** 用户已明确批准计划。

**角色转变：** 从架构师变为监督者。用户的提示变得极短。

### 执行原则

1. **实现计划中的一切** — 不精选，不跳过
2. **完成一个任务就在 plan.md 中标记** — 计划是进度的唯一来源
3. **不中断直到全部完成** — 不在途中暂停求确认
4. **保持代码干净** — 不添加不必要的注释或文档
5. **持续运行检查** — 编译检查、类型检查，尽早捕捉问题
6. **同步更新文档** — 任何确认的改动必须同步更新 plan.md 和 test-cases.md

### 文档同步规则

实现过程中，当发生以下情况时，必须同步更新 plan.md 和 test-cases.md：

| 触发条件 | plan.md 更新 | test-cases.md 更新 |
|----------|-------------|-------------------|
| 实现方案与计划有偏差 | 更新代码变更部分，标注实际方案 | 更新对应测试用例的预期结果 |
| 用户要求修改范围 | 增删相应 TODO 任务 | 增删对应测试用例 |
| 发现计划中的技术错误 | 纠正错误描述 | 调整测试步骤和预期 |
| 新增未预见的文件修改 | 补充到"将被修改的文件"列表 | 新增对应验证用例 |
| 接口签名变更 | 更新接口描述 | 更新接口兼容性测试 |

**同步格式：**
- plan.md 中用 `~~原文~~` + 新内容 标记变更，保留历史
- test-cases.md 中在"变更日志"部分记录每次修改

### 实现期间的用户反馈

实现阶段用户的反馈应该是简短的纠正：

- 遗漏纠正：`"你没有实现 deduplicateByTitle 函数"`
- 位置纠正：`"这个页面应该在 admin 模块，不是 main 模块，移过去"`
- 参考现有代码：`"这个表格应该和用户表完全一样，同样的表头、分页、行密度"`
- 视觉问题：`"更宽一点"` / `"还是被裁剪了"` / `"有 2px 的间距"`

### 方向错误时的处理

如果发现实现方向错误，不要试图修补。撤回并缩小范围：

> "我撤回了所有改动。现在我只想让列表视图更简洁 — 其他什么都不要改。"

撤回后缩小范围几乎总是比增量修复产生更好的结果。

### 保持控制权

即使在实现阶段，用户仍然掌握决策权：

- **精选建议：** "第一个用 Promise.all；第三个提取成函数；忽略第四和第五个"
- **修剪范围：** "从计划中删掉下载功能，现在不做"
- **保护接口：** "这三个函数的签名不能改，调用者来适配"
- **覆盖技术选择：** "用这个库的内置方法，不要自己写"

## 阶段四：审核文档 (Review) — Codex 执行 + 用户确认

**执行引擎：** Codex 官方插件（`codex:rescue`）

**触发时机：** Claude Code 完成全量实现后，**在测试验证之前**自动调用。

**目的：** 在正式验证之前，先让 Codex 对照最终的 plan.md、实际代码变更和 test-cases.md 生成一份审核文档清单，由用户确认实现是否符合预期。这是实现与验证之间的**质量关卡**。

### 4a：生成审核文档（Codex 执行）

**Claude Code 操作：**

```
codex:rescue 基于以下三份输入，生成审核文档清单：
1. <EPIC目录>/plan.md — 最终版实现计划（含所有注释循环的修改）
2. <EPIC目录>/test-cases.md — 测试用例文档
3. 实际代码变更 — 阅读本次实现涉及的所有修改文件

要求：
1. 只做审核分析，不修改项目代码文件
2. 将审核文档写入 <EPIC目录>/review.md
3. 审核维度：
   - 计划符合度：逐条对照 plan.md TODO 清单，检查每项是否已实现、实现是否与计划一致
   - 代码变更摘要：列出所有实际修改的文件，与 plan.md 中"将被修改的文件"对比，标记新增/遗漏
   - 偏差说明：实现与计划的偏差点，标注偏差原因（已在 plan.md 中记录的 vs 未记录的）
   - 测试覆盖度：对照 test-cases.md，检查每个测试用例是否有对应的实现支撑
   - 风险项：识别潜在风险（未覆盖的边界条件、可能的回归影响、接口兼容性问题）
   - 遗留问题：标记需要后续处理的事项
```

**审核文档结构：**

```markdown
# [功能名称] 审核文档

## 审核概览
- **审核日期：** YYYY-MM-DD
- **计划版本：** plan.md 最终版
- **实现状态：** 全部完成 / 部分完成

## 计划符合度

### TODO 清单逐条核查

| # | 任务描述 | 状态 | 备注 |
|---|---------|------|------|
| 1 | 任务描述 | ✅ 已实现 / ❌ 未实现 / ⚠️ 偏差 | 偏差说明 |
| 2 | ... | ... | ... |

**符合率：** X/Y (Z%)

## 代码变更摘要

### 计划内文件
| 文件路径 | 计划用途 | 实际变更 | 状态 |
|----------|---------|---------|------|
| path/to/file.java | 修改原因 | 实际做了什么 | ✅ 一致 / ⚠️ 偏差 |

### 计划外文件（实现中新增）
| 文件路径 | 变更原因 | 是否已更新 plan.md |
|----------|---------|-------------------|

### 未修改文件（计划中有但未改）
| 文件路径 | 计划用途 | 未修改原因 |
|----------|---------|-----------|

## 偏差说明
[列出所有实现与计划不一致的点，区分已记录偏差和未记录偏差]

## 测试覆盖度

| 测试用例 | 对应实现 | 覆盖状态 |
|----------|---------|---------|
| TC-1.1 | 功能描述 | ✅ 已覆盖 / ❌ 未覆盖 |

## 风险项
- [ ] 风险 1：描述 + 影响范围
- [ ] 风险 2：描述 + 影响范围

## 遗留问题
- [ ] 问题 1：描述 + 建议处理方式
- [ ] 问题 2：描述 + 建议处理方式

## 审核结论
[总体评价：建议通过 / 建议修复后通过 / 建议重新实现]
```

### 4b：用户审核确认

**交付：** 将 review.md 展示给用户，等待审查。

**用户审核要点：**
- 计划符合度是否达标？未实现的任务是否可接受？
- 偏差是否合理？未记录的偏差是否需要补充到 plan.md？
- 风险项是否可接受？是否需要在验证阶段重点关注？
- 遗留问题是否可以延后处理？

**用户反馈处理：**
- **审核通过** → 进入阶段五测试验证
- **需要修复** → 回到阶段三，针对审核中发现的问题进行修复，修复后重新生成 review.md
- **需要补充** → 用户在 review.md 中添加注释，Codex 更新审核文档

<HARD-GATE>
在用户明确确认审核文档通过之前，不能进入测试验证阶段。
</HARD-GATE>

## 阶段五：测试验证 (Verification) — Codex 执行

**执行引擎：** Codex 官方插件（`codex:rescue`）

**触发时机：** 用户确认 review.md 审核通过后，自动调用。

**Claude Code 操作：**

```
codex:rescue 对 <项目目录> 进行全面验证，基于 <EPIC目录>/test-cases.md 和 <EPIC目录>/review.md 进行逐条验证。

要求：
1. 只做检查和审查，不修改项目代码文件
2. 逐条执行 test-cases.md 中的测试用例，标记每条的通过/失败状态
3. 重点验证 review.md 中标记的风险项和偏差点
4. 编译检查：项目是否能正常编译
5. 测试执行：运行相关单元测试和集成测试
6. TODO 完成度：对照 plan.md 检查所有任务是否完成
7. 代码质量：检查明显的 Bug、安全问题
8. 集成验证：新代码与现有系统的兼容性
9. 将验证报告写入 <EPIC目录>/verification.md，包含每条测试用例的验证结果
```

**验证报告结构：**

```markdown
# [功能名称] 验证报告

## 总体结果
- 测试用例总数：X
- 通过：X
- 失败：X
- 未验证：X

## 审核风险项验证
[针对 review.md 中标记的风险项逐条验证]

| 风险项 | 验证结果 | 说明 |
|--------|---------|------|
| 风险 1 | ✅ 无问题 / ❌ 确认存在 | 详细说明 |

## 详细验证结果

### TC-1.1：[测试用例名称]
- **状态：** 通过 / 失败
- **验证方式：** [实际使用的验证方式]
- **备注：** [失败原因或注意事项]

## 编译检查
[编译结果]

## 代码质量
[发现的问题]

## TODO 完成度
[对照 plan.md 的检查结果]
```

**产出：** `verification.md` 写入 EPIC 目录

**后续处理：** Claude Code 读取 verification.md，向用户汇报结果。如有失败的测试用例或风险项确认存在，回到阶段三修复后重新走阶段四审核 → 阶段五验证。

## 核心原则

1. **Research 防止无知的改变** — 不理解系统就改代码，是最昂贵的失败模式
2. **Plan 防止错误的改变** — 计划是可编辑的持久化工件，不是转瞬即逝的聊天
3. **注释循环注入人类判断** — Claude 不知道产品优先级、用户痛点和工程权衡
4. **实现应该是无聊的** — 所有创意工作在计划阶段完成，执行只是机械操作
5. **审核防止遗漏和偏差** — 实现完成后先审核再验证，确保代码与计划一致，偏差被记录和确认
6. **单个长会话优于多个短会话** — 完整上下文胜过分割的会话，计划文档在压缩中保留完整保真度

## 反模式警告

- **跳过研究直接写计划** — 计划会基于错误假设，后续全部返工
- **注释循环中偷偷实现** — 破坏了"思考与打字分离"的核心原则
- **实现中做架构决策** — 所有决策应在计划阶段完成
- **修补而不是撤回** — 方向错误时，撤回+缩小范围比增量修复更好
- **跳过审核直接验证** — 没有审核就验证，等于不知道验证什么，偏差和遗漏会被忽略
